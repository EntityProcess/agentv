# Example YAML Config for Dolt Repo Sync

Here's an example YAML configuration for Dolt repository sync in `.beads/config.yaml`:

```yaml
# Dolt sync configuration
sync:
  remote: git+ssh://git@github.com/org/repo.git  # Primary Dolt remote URL

# Optional Dolt federation (for multi-peer sync)
federation:
  remote: dolthub://myorg/beads  # DoltHub or cloud remote
  sovereignty: T2  # Data sovereignty tier: T1, T2, T3, or T4

# Dolt history & sync behavior
dolt:
  auto-commit: on    # Create a Dolt commit after each successful write
  auto-push: false   # Opt-in for single-writer setups
  auto-push-interval: 5m   # Minimum time between auto-pushes
  auto-push-timeout: 30s   # Timeout for a single auto-push attempt
``` [1](#0-0) [2](#0-1) [3](#0-2) 

## Key Configuration Options

### Remote Configuration
- **`sync.remote`**: Primary Dolt remote URL (supports `dolthub://`, `gs://`, `s3://`, `az://`, `file://`, or git protocols like `git+ssh://`) [4](#0-3) 
- **`federation.remote`**: Dolt remote URL for federation setups [5](#0-4) 
- **`federation.sovereignty`**: Data sovereignty tier (`T1`-`T4`) controlling data residency restrictions [6](#0-5) 

### Sync Behavior
- **`dolt.auto-commit`**: Automatically create Dolt commits after write commands (default: `on`) [7](#0-6) 
- **`dolt.auto-push`**: Automatically push to Dolt remote after writes (opt-in, default: `false`) [8](#0-7) 
- **`dolt.auto-push-interval`**: Minimum time between auto-pushes (default: `5m`) [9](#0-8) 
- **`dolt.auto-push-timeout`**: Timeout for single auto-push attempt (default: `30s`) [10](#0-9) 

## Notes

The `sync.remote` key is a YAML-only configuration key that must be set in `config.yaml` rather than the database, as it's read during startup before the database is opened [11](#0-10) . For existing projects, you can also set the remote using `bd dolt remote add origin <url>` which will automatically write to `sync.remote` in the config file [12](#0-11) .

# Example Workspace/Repo Configuration

Here are example YAML configurations for Agent Orchestrator, ranging from minimal to comprehensive setups.

## Minimal Configuration

The absolute minimum required configuration:

```yaml
projects:
  my-app:
    repo: owner/my-app
    path: ~/my-app
    defaultBranch: main
``` [1](#0-0) 

## Basic Configuration with Defaults

A typical configuration with sensible defaults:

```yaml
# agent-orchestrator.yaml
$schema: https://raw.githubusercontent.com/ComposioHQ/agent-orchestrator/main/schema/config.schema.json
port: 3000

defaults:
  runtime: tmux       # default on macOS / Linux; on Windows the default is `process` (ConPTY)
  agent: claude-code
  workspace: worktree
  notifiers: [desktop]

projects:
  my-app:
    repo: owner/my-app
    path: ~/my-app
    defaultBranch: main
    sessionPrefix: app

reactions:
  ci-failed:
    auto: true
    action: send-to-agent
    retries: 2
  changes-requested:
    auto: true
    action: send-to-agent
    escalateAfter: 30m
  approved-and-green:
    auto: false # flip to true for auto-merge
    action: notify
``` [2](#0-1) 

## Full Configuration Example

The comprehensive example file with all available options:

```yaml
# Agent Orchestrator Configuration
# Copy to agent-orchestrator.yaml and customize.
$schema: https://raw.githubusercontent.com/ComposioHQ/agent-orchestrator/main/schema/config.schema.json

# Web dashboard port
port: 3000

# Terminal server ports
terminalPort: 14800
directTerminalPort: 14801

# Power management
power:
  preventIdleSleep: true  # Default on macOS, no-op on Linux

# Lifecycle controls
lifecycle:
  autoCleanupOnMerge: true
  mergeCleanupIdleGraceMs: 300000

# Default plugins
defaults:
  agent: claude-code
  workspace: worktree
  notifiers: [desktop]

projects:
  my-app:
    name: My App
    repo: org/my-app
    path: ~/my-app
    defaultBranch: main
    sessionPrefix: app

    # Per-project plugin overrides
    agent: claude-code
    runtime: tmux

    # Agent-specific config
    agentConfig:
      permissions: skip
      model: opus

    # Files to symlink into workspaces
    symlinks: [.env, .claude]

    # Commands to run after workspace creation
    postCreate:
      - "pnpm install"

    # Inline rules
    agentRules: |
      Always run tests before pushing.
      Use conventional commits (feat:, fix:, chore:).
``` [3](#0-2)

Based on the codebase context, you're asking for configuration examples that set up repository sync for checkpoint data. The Entire CLI uses JSON configuration files to sync checkpoint branches to a separate repository via the `checkpoint_remote` setting. [1](#0-0) 

## JSON Configuration Example

Here's a complete `settings.json` example that configures checkpoint sync to a separate repository:

```json
{
  "enabled": true,
  "strategy_options": {
    "checkpoint_remote": {
      "provider": "github",
      "repo": "myorg/checkpoints-private"
    }
  }
}
```

This configuration should be placed in `.entire/settings.json` in your repository root. [2](#0-1) 

## Configuration Details

The `checkpoint_remote` object requires two fields: [3](#0-2) 

- **`provider`**: The git hosting provider (e.g., "github")
- **`repo`**: The repository in "owner/repo" format (must contain a slash) [4](#0-3) 

## Alternative: CLI Configuration

You can also configure this via the CLI instead of editing the JSON file directly: [5](#0-4) 

```bash
entire enable --checkpoint-remote github:myorg/checkpoints-private
```

## How It Works

When configured, Entire will: [6](#0-5) 

1. Derive the checkpoint URL using the same protocol (SSH or HTTPS) as your push remote
2. Push the `entire/checkpoints/v1` branch to the specified checkpoint repository instead of your default push remote
3. Skip pushing if a fork is detected (when push remote owner differs from checkpoint repo owner) [7](#0-6) 
4. Fetch the checkpoint branch locally if it exists on the remote but not locally (one-time setup) [8](#0-7) 

## Notes

- The configuration only supports JSON format, not YAML, based on the codebase implementation
- The `repo` field must contain a slash (e.g., "org/repo") or the configuration will be rejected [4](#0-3) 
- Legacy string format (e.g., `"checkpoint_remote": "git@github.com:org/repo.git"`) is no longer supported and will be ignored [9](#0-8)
